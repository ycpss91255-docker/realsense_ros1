# RealSense Dynamic Calibration Tool

このページでは、**Intel RealSense D400 Series Dynamic Calibration Tool**
(`librscalibrationtool`) について説明します。すなわち、このツールが何をするのか、
`CAMERA.md` にあるオンチップキャリブレーションとどう違うのか、そしてどのように
実行するのかを解説します。

> **提供状況に関する注記 (ROS 1 / focal):** このツールは現時点では
> **このリポジトリの `devel` イメージには同梱されていません。** ROS 2 版
> (`realsense_ros2`) では Intel の `pool/jammy` `.deb` から同梱していますが、
> そのパッケージは **amd64 専用であり Ubuntu リリースに紐付いており**、この
> リポジトリのベースは **Ubuntu 20.04 focal** (ROS 1 Noetic) です。focal 互換の
> ビルドの同梱は後続の作業に先送りされています。それまでは、ツールをホスト上で
> (または手動でインストールした focal コンテナ内で) 実行してください。以下の
> ワークフローは、ツールが利用可能になれば引き続き適用されます。

## 概要 (オンチップキャリブレーションとの違い)

D400 カメラには、2 つの異なるキャリブレーション経路があります:

| | オンチップキャリブレーション | Dynamic Calibration Tool |
|---|---|---|
| 掲載箇所 | [CAMERA.md](CAMERA.md) (セクション 5) | このページ |
| 実行元 | `realsense-viewer` (内蔵) | `Intel.Realsense.DynamicCalibrator` |
| ターゲットの要否 | 不要 -- テクスチャのある任意のシーン | 必要 -- 印刷した (またはスマホアプリの) ターゲット |
| キャリブレーション対象 | 深度の整流のみ (ステレオ IR) | 整流 + 深度スケール、**加えて** RGB デバイスでは **RGB の外部パラメータ** |
| 使用する場面 | 深度ノイズ / 平面が平らでない場合 | ターゲットに基づく徹底した再キャリブレーションが必要な場合 |

Dynamic calibration が最適化するのは **外部 (extrinsic)** パラメータのみ、すなわち
イメージャ間の回転と並進であり、内部パラメータ (焦点距離、主点、歪み) は工場出荷時の
キャリブレーションのまま維持されます。User Guide (v2.11) によれば、2 つの動作モード
(targeted と target-less) で 2 種類のキャリブレーションを提供します。
`Intel.Realsense.DynamicCalibrator` の GUI/CLI は **targeted** キャリブレーションを
実行します:

- **Rectification calibration (整流キャリブレーション)** -- 2 つの IR イメージャの
  エピポーラ幾何を再整列します (`RotationLeftRight` / `TranslationLeftRight`)。
  オンチップキャリブレーションと同じ目的ですが、ターゲットに基づくものです。
- **Depth scale calibration (深度スケールキャリブレーション)** -- 光学素子が
  ずれた場合に、絶対深度スケールを補正します。

D455 における Depth<->RGB: targeted キャリブレーションは、RGB センサを持つデバイス
(D415/D435/D455) では **RGB の外部パラメータも再キャリブレーションします**
(`RotationLeftRGB` / `TranslationLeftRGB` -- 左イメージャに対するカラーセンサの関係)。
これはまさに、ランタイムのアライメントが依拠する depth-to-color の関係
(`realsense2_camera` の `align_depth:=true`、または SDK の `rs2::align`) であり、
再実行することで depth<->color オーバーレイのずれが修正されます。target-less モードは
深度 (左/右) のみをキャリブレーションし、RGB は **行いません**。また API 専用であり、
GUI キャリブレータでは提供されません。

## ツールが提供するもの

Intel はこのツールを、事前コンパイル済みの **amd64 専用** パッケージとして提供します
(ARM64 ビルドはありません)。このパッケージは以下の実行ファイルを提供します:

| 実行ファイル | 用途 |
|---|---|
| `Intel.Realsense.DynamicCalibrator` | Targeted dynamic calibration (整流 + 深度スケール + RGB 外部パラメータ)、GUI および CLI |
| `Intel.Realsense.CustomRW` | カメラに保存されたキャリブレーションテーブルの読み取り / 書き込み |
| `opencv_interactive-calibration` | OpenCV インタラクティブキャリブレーションのヘルパー |

## 前提条件

1. **ホストの udev ルール。** このツールはカメラへの生の USB アクセスを必要とし、
   これは SDK の他の部分と同じ権限要件です。ホスト上で一度だけインストールします:

   ```bash
   ./script/install_udev_rules.sh
   ```

   これがコンテナ内だけでなくホスト上になければならない理由については、README の
   「RealSense udev Rules」セクションを参照してください。

2. **キャリブレーションターゲット。** 公式ターゲットを規定のスケールで印刷する
   (下記リンク) か、Intel RealSense Dynamic Target スマホアプリ (iOS / Android) で
   表示します。

3. **GUI アクセス。** キャリブレータのウィンドウを開けるように、X11/Qt/OpenGL
   スタックを備えたディスプレイが必要です (`devel` イメージにはこれが含まれており、
   GUI モードで動作します)。

## 実行方法

```bash
# Once the tool is available on PATH (see the availability note above):
Intel.Realsense.DynamicCalibrator     # interactive GUI calibration
```

デバイスをターゲットから **600--850 mm** の位置に配置し、ターゲットのバーが視野内で
おおよそ垂直になるようにします。全過程を通じてカメラとターゲットの相対的な動きが
必要です (一方を固定し、もう一方を動かす)。反射 (日光、明るい照明、スマホ画面の
まぶしさ) は避けてください -- ターゲットの検出を妨げます。targeted のフローは、その後
以下のフェーズを順番に実行します (User Guide セクション 4.5.4 および Appendix B):

1. **Rectification phase (整流フェーズ)** -- ライブビューの中央に、影付きの
   **青い四角** のブロックが重ねて表示されます。各青い四角は、まだターゲットの
   カバーが必要な視野内の領域を示します。カメラ (またはターゲット) をゆっくり
   動かして、ターゲットの黒/白の四角やバーが青い四角に重なるようにすると、
   カバーされた四角が 1 つずつ **消えて** いきます。すべて消えるまで繰り返します。
   (オートエクスポーズの探索がオンでターゲットが一時的に見失われた場合、探索中に
   画像が明<->暗と切り替わります -- これは想定通りです。まったく検出されない場合は、
   反射や距離を修正するように位置を調整してください。) 中間結果は直ちにストリームに
   適用されます。
2. **Scale phase (スケールフェーズ)** -- 自動的に開始します。ターゲットを異なる
   別々の位置へ配置し直し続け、**15** 枚のターゲット画像が受け入れられる
   (緑色のプログレスバーが完了まで満たされる) まで続けます。
3. **RGB phase (RGB フェーズ)** (RGB デバイスのみ -- D415/D435/D455) -- スケール
   フェーズと同様に、15 枚のターゲット画像をキャプチャし、depth-to-RGB の UV
   マッピングをキャリブレーションします。これが終わると、左/右の深度と depth<->RGB の
   両方がキャリブレーションされます。

完了すると、結果はカメラに書き込まれます。前後でキャリブレーションテーブルを
バックアップまたは復元するには `Intel.Realsense.CustomRW` を使い、その後に深度品質を
検証してください (満足できない場合は再実行します)。

## 既知の制限: depth<->color アライメントの残差誤差

キャリブレーションが成功した後でも、depth-to-color オーバーレイ (`align`) には
まだいくらかの残差誤差があり、これはオブジェクトのエッジ付近で最も目立ちます。
ハードウェアで検証済み: **D455 では ~1--2 m で明らかに目立ち**、**D435 では
わずかに存在** します。これは大部分が **想定通りかつ幾何学的なもの** であり、
キャリブレーションが失敗した兆候ではありません。キャリブレーションは系統的な
外部オフセットを除去しますが、以下は除去できません:

- **視差 / オクルージョン** -- 深度 (左 IR) と RGB は光学中心が異なるため、
  オブジェクトの境界では一方のカメラに見えるものがもう一方には見えません。この領域は
  いかなるキャリブレーションでもアライメントできません -- これは純粋な幾何であり、
  エッジの「フリンジ (縁取り)」の主な原因です。
- **深度誤差** -- ステレオ深度の誤差はおおよそ距離の 2 乗で増大するため、1--2 m では
  カラー画像への逆投影 (deprojection) の精度が低下します (ノイズの多いエッジや穴で
  さらに悪化します)。
- **RGB のローリングシャッター / 同期** -- カラーセンサはローリングシャッターであり、
  カメラやシーンの動きがあると (グローバルシャッターの) 深度フレームに対してずれます。

D455 が D435 より悪い理由: **D455 は 95 mm のステレオベースラインを持つのに対し、
D435 は 50 mm** です。ベースラインが広いほど長距離の深度は良くなりますが、depth<->RGB
の視差が大きくなるため、近距離/中距離で残差がより目立ちます。

依然として有効な対策 (ゼロにはなりません):

- ユースケースに応じて、depth->color と color->depth のどちらでアライメントするかを
  選択します。
- アライメントの *前に* 深度の後処理 (spatial / temporal / hole-filling) を適用します。
- 深度/カラーの同期を保ち、ローリングシャッターのずれを避けるために静止シーンを
  撮影します。
- 深度ができるだけ正確になるよう、カメラの最適な深度レンジ内に収めます。

### アライメント方向: depth->color と color->depth

この 2 つの方向は **対称ではなく**、異なるアルゴリズムです。これが上記の選択が重要である
理由です (Intel の *Projection, Texture-Mapping and Occlusion* ホワイトペーパー、
セクション 3.4 より):

- **Color->depth** (カラーを深度フレームに合わせる) は安価な方向です。各深度ピクセルに
  ついて、その uv-map 座標を参照し、そこのカラーを取得します。出力は深度解像度で、
  1 ピクセルあたり 1 回の参照 -- 穴はできません。
- **Depth->color** (深度をカラーフレームに合わせる -- `align_depth:=true` /
  `rs_aligned_depth.launch` が生成するもの) は「少し厄介」です。カラーピクセルは
  容易には 3D に逆投影できないため、代わりにすべての深度ピクセルをカラー座標へ
  **順投影 (forward-projection)** し、複数が同じカラーピクセルに落ちた場合は最も近い
  深度を残します (z-buffering)。カラーセンサが深度より高解像度の場合、一部のカラー
  ピクセルには深度が割り当てられず -> 穴ができ、SDK は各深度ピクセルを 2x2 のパッチに
  スプラット (splat) することでこれを埋めます。

したがって、出荷時のデフォルト (`depth->color`) はより難しく近似に陥りやすい方向です
-- 順投影 + 2x2 の穴埋めが、残差がエッジのフリンジとして最も目立つ理由の一部です。
深度解像度の出力を扱えるアプリケーションは、`color->depth` を選ぶことでこの種の
アーティファクトを回避できます。

## 公式ドキュメントが定量化しているもの (と、していないもの)

Intel は、距離に対する depth<->color の *アライメント* 誤差の表を **公開していません**。
最も近い公式の数値は以下です:

- **深度 Z 精度 (絶対誤差): +/-2%** -- D400-Series Datasheet (337029-017) Table 4-15
  「Depth Quality Specification」に、Fill Rate >=99%、RMS Error <=2%、Temporal Noise
  <=1% と併記。測定距離はモデルによって異なり、D43x の仕様は「<=2 m, 80% ROI, HD」、
  D450/D455 の仕様は「<=4 m, 80% ROI, HD」と記載されています。**これは深度精度の仕様で
  あり、アライメント誤差の仕様ではありません** -- Z 値がどれだけ誤っているかを規定する
  もので、それはカラーへの逆投影に *間接的* に影響するだけです。距離ごと (1 m / 2 m /
  3 m) の内訳はありません。
- **アライメントアルゴリズムとその誤差要因** は、*Projection, Texture-Mapping and
  Occlusion* ホワイトペーパーで *定性的* に記述されています: セクション 3.4 (ストリーム
  アライメント -- 上記の 2 つの方向) とセクション 4 (オクルージョン無効化 -- オブジェクト
  境界では一方のイメージャに見えるものがもう一方には見えず、SDK 2.35+ は点群計算中に
  これらのオクルージョンされた領域を自動的に無効化します)。エッジ誤差が *なぜ* 存在するか
  は説明していますが、測定された大きさは示していません。

結論: 「1 m / 2 m / 3 m の残差」の数値は **Intel からは入手できず** -- 自前で測定する
必要があります (下記参照)。

## 定量化実験の計画 (先送り)

目標: 「D455 は 1--2 m で悪化する」という定性的な観察を、距離対誤差の表 (1 m / 2 m /
3 m での誤差) に変換すること。

**先送り (2026-07-03)。** 信頼できる測定には **安定した制御された構成** が必要ですが、
現在利用可能な唯一の D455 は稼働中の AMR (ナビゲーションスタックを実行中) に取り付け
られており、カメラ姿勢が制御されていません -- 測定用リグではなく本番ノードです。専用の
静的な構成が利用可能になったら、これを一度実施します。

設計:

- **指標 (主): edge-offset (エッジオフセット)。** 既知の直線エッジを持つオブジェクト
  (平らな板、またはチェッカーボード) にカメラを向けます。アライメント済みフレームで
  **深度エッジ** と **カラーエッジ** の間のピクセルオフセットを測定し、深度 Z とカメラ
  内部パラメータを介して mm に変換します。当初観察されたエッジのフリンジに対応し、
  「1 m ではエッジが X px / Y mm ずれている」と直接読み取れます。
- **指標 (任意、より厳密): checkerboard reprojection (チェッカーボード再投影)。**
  カラー内のチェッカーボードのコーナーと、深度から逆投影したコーナーを検出し、再投影
  誤差 (px / mm) を報告します。数値に説得力が必要な場合 (レポート / 外部向け) に使用します。
- **アライメント方向:** 出荷デフォルトである **depth->color** (`align_depth:=true`) を
  測定し、数値が出荷内容に対応するようにします。任意で比較のため `color->depth` も測定
  します。
- **キャプチャ:** 1 / 2 / 3 m のそれぞれで (カメラのカバーガラス -- Ground-Zero 基準、
  datasheet 4.8 -- を基準に測定)、同期したアライメント済みの深度+カラーを、静止シーンで、
  カメラをロボットから外した状態で記録します。本番と同様に、アライメントの *前に* 深度の
  後処理 (spatial / temporal / hole-fill) を適用します。
- **対照:** ベースラインに起因する差 (95 mm 対 50 mm) を再現するため、D455 と D435 の
  両方を実行します。ターゲットは十分な照明で、反射なく、おおよそ ROI を満たすようにします。
- **出力:** 距離対誤差の表 + オーバーレイのスクリーンショットを、ここに追記します。

前提条件: 実行中のコンテナが実際に
`/camera/aligned_depth_to_color/image_raw` を publish している必要があります。2026-07-03
より前にビルドされたランタイムイメージは、非アライメントの `rs_camera.launch` を
デフォルトとします。新しいアライメントのデフォルトを取り込むために再ビルドするか、
キャプチャのためにアライメントを明示的に起動してください。
